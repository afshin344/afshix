the problem

Designing within someone else's system, not from a blank canvas

Unlike a greenfield project, Houmify's core visual system already existed — so the job wasn't "design a real‑estate marketplace," it was "improve the flows and add new capability without breaking what the platform already established." That's a different, more constrained kind of design problem: every decision had to align with existing patterns before it could improve on them.

process

Redesigning flows while respecting the existing system

I audited the existing user flows for finding agents and listings, identified where users were dropping off or getting confused, and redesigned those specific paths while keeping visual and interaction consistency with the rest of the platform. I also developed a prototype that accounted for real backend and database constraints — so what shipped wasn't just visually correct but technically buildable from day one.

key decisions

Where the trade‑offs actually happened

decision — extend the system, don't replace it

Rather than proposing a visual overhaul, I worked within Houmify's existing component patterns — the faster, lower‑risk path for a live marketplace, and the one that let engineering ship incrementally instead of a disruptive relaunch.

decision — an AI assistant as a shortcut, not a replacement

The AI listings assistant was scoped to help users find relevant listings faster — a layer on top of the existing search and browse flows, not a replacement for the marketplace's core discovery mechanics that agents and owners already relied on.

the solution

Improved flows, plus an AI shortcut to the right listing

The delivered redesign kept Houmify's platform identity intact while closing the specific friction points in agent discovery and listing search — with an AI assistant layered in to help users cut through the marketplace faster.

reflection

What I'd revisit

Staying inside the existing design system was the right call for shipping speed, but a couple of legacy patterns I chose not to touch are probably due for a proper rework rather than another workaround — that's a conversation for a future phase, not this one.
